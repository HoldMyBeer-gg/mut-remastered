# Plan 03-01: Character Creation and Database Schema

**Phase:** 03-character-and-combat
**Plan:** 1 of 3
**Scope:** AUTH-03, AUTH-04, AUTH-05, AUTH-06, AUTH-07

## Goal

A player can register, log in, create a character with race/class/ability scores/gender, select that character, and enter the world. Multiple characters per account are supported. The existing player_positions system migrates from account_id to character_id.

## Prerequisites

- Phase 2 complete (world + movement + auth working)
- All existing tests pass

## Task 1: Protocol Messages, Character Types, and DB Schema

### New Files

**`protocol/src/character.rs`** — Character-related message types:
```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
pub enum ClientMsg {
    /// List all characters on the current account
    CharacterList,
    /// Create a new character
    CharacterCreate {
        name: String,
        race: String,
        class: String,
        gender: String,
        /// Ability scores: [STR, DEX, CON, INT, WIS, CHA]
        ability_scores: [u8; 6],
        /// For Human and Half-Elf: which abilities get their racial "free" points
        /// Human: two indices (0-5) for +1 each
        /// Half-Elf: one index (0-5) for +1
        /// Other races: ignored
        racial_bonus_choices: Vec<u8>,
    },
    /// Select an existing character to play
    CharacterSelect { character_id: String },
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
pub enum ServerMsg {
    CharacterListResult {
        characters: Vec<CharacterSummary>,
    },
    CharacterCreateOk {
        character_id: String,
        name: String,
    },
    CharacterCreateFail {
        reason: String,
    },
    CharacterSelected {
        character_id: String,
        name: String,
    },
    CharacterSelectFail {
        reason: String,
    },
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
pub struct CharacterSummary {
    pub id: String,
    pub name: String,
    pub race: String,
    pub class: String,
    pub level: u32,
}
```

## Task 2: ConnectionActor Character Flow and DB Handlers

### Modified Files

**`server/src/session/actor.rs`** — Extend ConnectionActor:

Add new fields:
```rust
/// Active character ID (set after CharacterSelect)
active_character_id: Option<String>,
/// Active character name (for display in room/broadcasts)
active_character_name: Option<String>,
```

Add new state machine: After login, the player is in "character select" state. World commands are rejected until a character is selected.

Add `dispatch_frame` handling for NS_CHAR namespace:
```rust
if let Ok(msg) = decode_message::<protocol::character::ClientMsg>(NS_CHAR, bytes) {
    return self.handle_character_message(msg).await;
}
```

Implement `handle_character_message`:
- `CharacterList` → query DB for account's characters, return `CharacterListResult`
- `CharacterCreate` → validate inputs (race, class, gender strings; point-buy; name uniqueness), insert into DB, return `CharacterCreateOk` or `CharacterCreateFail`
- `CharacterSelect` → verify character belongs to account, load from DB, place in world (using `character_positions` table), subscribe to room, return `CharacterSelected` + `Vitals` + `RoomDescription`

Modify `handle_world_message` to use `active_character_id` instead of `account_id` for:
- Player position lookups
- Room broadcasts (use character name instead of account_id)
- Position persistence

Modify `ensure_player_in_world` → `ensure_character_in_world` using character_id.

Modify `cleanup` to handle character logout (save position, remove from in-memory world).

**`server/src/world/types.rs`** — Update World struct:
- `player_positions: HashMap<String, RoomId>` — key changes from account_id to character_id
- Add `player_names: HashMap<String, String>` — character_id → character_name for display

**`server/src/world/commands.rs`** — Update all command handlers:
- Replace `account_id` parameter with `character_id` 
- Use `player_names` for broadcast messages (show character name, not account_id)
- `handle_look` shows character names in `players_here` instead of account IDs

**`server/src/net/listener.rs`** — No changes needed to AppState (character data is in DB, loaded on select)
